### 🔬 ГРУППА 1 — Борьба с Overfitting (из поиска по backtesting/p-value)

---

**1. GAN-синтетические данные для anti-overfitting теста**
*[arXiv: Backtesting Trading Strategies with GAN To Avoid Overfitting]*

Основная идея статьи: хорошая стратегия должна работать не только на реальных исторических данных, но и на синтетических путях, сгенерированных GAN-сетью, которая обучилась воспроизводить распределение реальных данных. Применительно к USE Optimizer: после нахождения топ-стратегии запустить её бэктест на N=100 синтетических equity путях (имитирующих статистику реальных данных) и посчитать % прогонов с положительным PnL. Стратегия с 90%+ = робастная. Это качественно сильнее MC Permutation, потому что сохраняет не только распределение сделок, но и серийные зависимости в ценах.

---

**2. Covariance Penalty для отбора параметров**
*[arXiv: Avoiding Backtesting Overfitting by Covariance-Penalties]*

Статья предлагает штрафовать за correlation между стратегиями: если в топ-100 результатов много похожих (высокая корреляция equity curves) — вычитать "overfitting penalty". Идея для оптимизатора: после оптимизации выбирать финальный набор результатов не только по PnL/GT, но и отдавая предпочтение некоррелированным стратегиям. Пользователь получает "портфель 10 непохожих стратегий" вместо "10 вариаций одного и того же".

---

**3. GT-Score с академической базой (из найденной статьи)**
*[arXiv: The GT-Score: A Robust Objective Function for Reducing Overfitting]*

Статья на arXiv исследует именно GT-Score как анти-overfitting метрику! Нужно прочитать её полностью и сравнить с нашей реализацией `_calcGTScore()`. Возможно академическая версия использует другие коэффициенты или дополнительные нормализации. Если есть расхождения — обновить нашу формулу по статье. Это укрепит математическое обоснование метрики.

---

**4. Информационный критерий для числа параметров (AIC/BIC адаптация)**
*[arXiv: Hyperparameter Optimization in Machine Learning — AIC/BIC section]*

Akaike Information Criterion: `AIC = 2k − 2ln(L)` где k = число параметров, L = правдоподобие. Адаптация для трейдинга: штрафовать за каждый дополнительно включённый фильтр или параметр. Стратегия с 10 фильтрами и WR 60% хуже чем с 3 фильтрами и WR 58%, если AIC учитывается. Добавить как опциональный метод ранжирования.

---

**5. Правило Минимума Описания (MDL — Minimum Description Length)**
*[arXiv: Hyperparameter Optimization — model selection section]*

MDL: лучшая стратегия — та, которая "описывает" данные наиболее компактно. Количество параметров × сложность комбинации / качество результата. Строгая математическая альтернатива AIC/BIC. В контексте оптимизатора: стратегия с 2 фильтрами и GT=5 предпочтительнее стратегии с 8 фильтрами и GT=5.1. Реализация: добавить поле "complexity" и метрику MDL-Score в таблицу.

---

**6. Permutation Feature Importance для фильтров**
*[arXiv: Backtesting with GAN — feature importance раздел]*

По аналогии с ML: взять готовую стратегию, случайно перемешать значения одного фильтра (например, рандомизировать RSI) и посмотреть насколько упал PnL. Если RSI-фильтр важен — PnL падает сильно. Если нет — его можно убрать без потери качества. Реализуется через 100 permutation-прогонов на каждый активный фильтр. Показывает пользователю "карту важности" каждого компонента стратегии.

---

**7. Walk-Forward с интерпретируемыми гипотезами**
*[arXiv: Interpretable Hypothesis-Driven Trading: A Rigorous Walk-Forward Validation Framework]*

Статья вводит принцип: перед оптимизацией формулируй явную гипотезу ("RSI oversold + тренд = хорошая точка входа"), оптимизируй только параметры этой гипотезы, валидируй на walk-forward. Без явной гипотезы — data mining. Для USE Optimizer: добавить поле "гипотеза" в шаблон, которое пользователь заполняет текстом. Агент может автоматически предлагать гипотезы для найденных стратегий.

---

**8. Минимальный трек-рекорд (Minimum Track Record Length)**
*[arXiv: Backtesting strategies with GAN — MLT section]*

Bailey & López de Prado: сколько лет данных нужно чтобы стратегия с Sharpe X была статистически значимой? Формула зависит от Sharpe и его дисперсии. Например, при Sharpe=1 нужно минимум 4.5 года. Добавить предупреждение: "Ваши данные охватывают X лет. Для надёжности нужно минимум Y лет." Расчёт займёт ~5 строк.

### 📊 ГРУППА 2 — Метрики качества (из поиска по Sharpe/Sortino/Calmar)

---

**9. Robust Sharpe Ratio (с поправкой на fat tails)**
*[arXiv: Financial metrics like the Sharpe ratio — arXiv:2502.00029]*

Классический Sharpe предполагает нормальное распределение доходностей. Реальные трейдинговые стратегии имеют "жирные хвосты" (редкие большие потери). Robust Sharpe поправляет стандартное отклонение на skewness и kurtosis распределения сделок. Формула: `Sharpe_robust = Sharpe × (1 - skew/6 × Sharpe - (kurtosis-3)/24 × Sharpe²)`. Нужен `tradePnl[]`.

---

**10. AlphaSharpe — LLM-оценка стратегии**
*[arXiv: AlphaSharpe: LLM-Driven Discovery of Robust Risk-Adjusted Metrics]*

Статья предлагает использовать LLM для автоматического поиска новых метрик качества, лучше Sharpe. Алгоритм: LLM генерирует кандидатов метрик, эволюционный алгоритм отбирает лучших. Для USE Optimizer: идея применения — дать агенту задачу "найти лучшую комбинацию существующих метрик (UPI × CVR × Sig%)". Агент оптимизирует веса комбинации по корреляции с реальной OOS performance.

---

**11. Sortino с порогом доходности (не нулём)**
*[arXiv: Optimal allocation using the Sortino ratio]*

Стандартный Sortino штрафует за доходности ниже нуля. Статья показывает, что оптимальный вариант — штрафовать за доходности ниже risk-free rate или MAR (Minimum Acceptable Return). Если пользователь хочет доходность минимум 20% в год — Sortino должен считаться относительно этого порога. Добавить параметр "target return" в настройки метрики.

---

**12. Information Ratio (IR)**
*[arXiv: arXiv:2405.05449 — Performance metrics comparison table]*

`IR = (annualized_return − benchmark_return) / tracking_error`. Если бэктест по сути сравнивается с buy-and-hold (что часто верно), IR показывает реальную ценность стратегии над рынком. Tracking error = стандартное отклонение разности доходностей стратегии и бенчмарка. Требует бенчмарк-данных (equity DATA как proxy).

---

**13. Burke Ratio — улучшенный Calmar**
*[arXiv: Hierarchical Risk Parity for Portfolio — metrics comparison]*

`Burke = (annualized return) / sqrt(sum(DD_i²))` где DD_i — каждая отдельная просадка (не только максимальная). Лучше Calmar потому что учитывает частоту просадок, не только глубину. Лучше UPI потому что считает по отдельным просадкам, а не по equity bars. Нужна функция нахождения всех просадок из equity curve.

---

**14. Serenity Index (комплексная метрика)**
*[arXiv: arXiv:2502.00029 — metrics beyond Sharpe]*

`Serenity = (total_return) / (VaR × DurationOfDrawdown)`. Штрафует и за глубину потерь и за время восстановления. Трейдеры часто говорят "меня не убивает просадка — меня убивает её длительность". Serenity формализует это интуитивное ощущение. Расчёт: нужны equity bars для нахождения длительности каждой просадки.

---

**15. Ω-Sharpe (Omega ratio как замена дисперсии)**
*[arXiv: AlphaSharpe — нестандартные метрики risk-adjusted return]*

Вместо `mean/std` (Sharpe) использовать `mean / (downside_std + 1/Omega)` — комбинация, учитывающая и стабильность потерь и соотношение выигрышей/проигрышей. Omega уже реализован в проекте. Задача: создать составную метрику объединяющую Sortino + Omega. Исследовать коэффициенты на исторических данных проекта.

---

**16. Probabilistic Sharpe Ratio (PSR)**
*[arXiv: AlphaSharpe — López de Prado reference]*

`PSR = Φ(√(n-1) × (SR - SR*) / √(1 - skew×SR + (kurtosis-1)/4 × SR²))` где SR* = минимальный разумный Sharpe (например 1.0). Показывает вероятность что наш Sharpe лучше SR*. PSR=95% = "мы на 95% уверены что SR реально > 1". Требует skewness и kurtosis `tradePnl[]`, ~20 строк.

### 🔄 ГРУППА 3 — Walk-Forward и кросс-валидация

---

**17. Combinatorial Purged CV (CPCV v2 — академическая версия)**
*[arXiv: Interpretable Hypothesis-Driven Trading — CPCV reference]*

Наш CPCV делает 5 блоков, каждый тестируется независимо. Академическая версия (Bailey et al.) добавляет "purging" и "embargo" — удаление баров вблизи границ блоков чтобы устранить lookahead bias через корреляцию соседних баров. Без embargo: последний бар IS-блока коррелирован с первым баром OOS-блока → завышенные OOS метрики. Добавить параметр `embargo_bars` в `_calcCPCVScore()`.

---

**18. Optuna-стиль TPE с многократным перезапуском**
*[arXiv: XGBoost Forecasting with Walk Forward Validation — Optuna hyperparameter section]*

Статья использует Optuna (TPE алгоритм) с многократными перезапусками из разных случайных точек для избежания локальных максимумов. В нашем TPE: добавить опцию N random restarts — каждые K итераций обнуляем TPE историю и стартуем с новой случайной точки. Финально берём лучший результат из всех стартов. Это стандартная практика в индустрии.

---

**19. Multi-Scale Walk-Forward (разные горизонты)**
*[arXiv: Interpretable Hypothesis-Driven Trading — раздел multi-horizon]*

Запускать walk-forward не на одном OOS горизонте, а на нескольких: 10%, 20%, 30% OOS. Стратегия хороша только если работает на всех горизонтах. Отображать: "OOS 10%: +12%, OOS 20%: +8%, OOS 30%: +5%" — трейдер видит тренд деградации. Сейчас только один split (70/30).

---

**20. Стресс-тест на "кризисных" сегментах**
*[arXiv: Market Regime Detection via Realized Covariances — режим "crisis"]*

Статья выделяет "кризисный режим" как особый рыночный режим с высокой волатильностью и скачками корреляций. Для USE Optimizer: детектировать сегменты с аномально высоким ATR (> 95-й персентиль исторического ATR) и тестировать стратегию отдельно на этих сегментах. Это кризисный стресс-тест: "как стратегия ведёт себя во время обвалов".

### 🎯 ГРУППА 4 — Новые входы (из поиска по техническим индикаторам и свечам)

---

**21. Bollinger Band Squeeze с Keltner Channel**
*[arXiv: Assessing the Impact of Technical Indicators on Machine Learning Models]*

Статья находит что BB + Keltner Channel squeeze (BB внутри KC) — один из сильнейших индикаторов консолидации. Когда BB narrower чем KC → squeeze. При выходе BB из KC с направлением → сильный сигнал. Отличается от простого BB squeeze: требует одновременно два индикатора. Реализация: BB = SMA ± 2σ, KC = EMA ± ATR×1.5.

---

**22. Elder Impulse System (EIS)**
*[arXiv: Technical Indicator Networks — TINs framework]*

Комбинация EMA тренда и MACD гистограммы: если EMA растёт И MACD гистограмма растёт → сильный бычий импульс. Вход только когда оба согласованы. Статья находит это одной из самых интерпретируемых комбинаций. calcEMA и calcMACD уже есть. Нужен только entry в реестр с проверкой двух условий одновременно.

---

**23. Triple Exponential Moving Average (TEMA)**
*[arXiv: Assessing the Impact of Technical Indicators — comparison table]*

`TEMA = 3×EMA1 − 3×EMA2 + EMA3` где EMA2 = EMA(EMA1), EMA3 = EMA(EMA2). Значительно снижает лаг по сравнению с обычной EMA при сохранении гладкости. Статья показывает TEMA outperforms SMA/EMA как фильтр тренда. Реализация: 3 прохода calcEMA и формула. ~10 строк. Добавить как опцию типа MA в фильтре.

---

**24. DEMA (Double EMA) как фильтр**
*[arXiv: Assessing the Impact of Technical Indicators — DEMA section]*

`DEMA = 2×EMA − EMA(EMA)`. Проще TEMA, но тоже снижает лаг. Статья сравнивает: на трендовых рынках DEMA лучше простой EMA, на боковых — хуже. Интересная идея: автоматически переключаться между EMA и DEMA в зависимости от ADX (ADX > 25 = тренд = DEMA, ADX < 20 = боковик = EMA).

---

**25. Распознавание свечных паттернов через условия (без CV)**
*[arXiv: Stock Chart Pattern recognition with Deep Learning — traditional conditions раздел]*

Статья сравнивает CNN-распознавание паттернов с традиционными условиями (if/else) и показывает: для базовых паттернов (Engulfing, Pin Bar, Doji, Inside Bar, Hammer) традиционные условия не хуже ML при правильной параметризации. Задача: добавить 3-5 новых свечных паттернов используя явные условия, аналогично существующим Engulfing и PinBar. Hammer (молот), Shooting Star, Three White Soldiers.

---

**26. Паттерн "Three White Soldiers / Three Black Crows"**
*[arXiv: Using Deep Learning Neural Networks and Candlestick Chart]*

3 последовательные бычьи (или медвежьи) свечи с закрытием в верхней/нижней части диапазона, каждая открывается внутри тела предыдущей. Статья находит этот паттерн с точностью ~65% на дневных данных. Реализация: проверка 3 свечей назад с условиями на тело и открытие. ~15 строк. Хорошо сочетается с фильтром Struct (рыночная структура).

---

**27. Parabolic SAR как exit-триггер**
*[arXiv: Technical Indicator Networks — TINs, exit signals section]*

Parabolic SAR даёт точки разворота: при длинной позиции — если SAR превысил цену → выход. Статья использует его как exit-сигнал, а не как entry. В EXIT_REGISTRY добавить проверку: SAR перешёл уровень цены → выход по close. calcSAR: ~20 строк (стандартный алгоритм с acceleration factor).

---

**28. RSI Divergence (бычье/медвежье расхождение)**
*[arXiv: Assessing the Impact of Technical Indicators — divergence section]*

Классическое расхождение: цена делает новый минимум, но RSI — нет (бычья дивергенция). Статья подтверждает: RSI divergence имеет статистически значимое предсказательное значение при правильной реализации (с точным определением свингов). Нужен поиск последних 2 pivot highs/lows — уже есть в core.js. Сравниваем цену и RSI в этих точках.

---

**29. MACD как фильтр (не только как entry)**
*[arXiv: Technical Indicator Networks — multi-signal ensemble]*

Статья выявляет: MACD лучше работает как фильтр (блокировать торговлю против MACD тренда), чем как entry-сигнал. Добавить MACD в FILTER_REGISTRY: если MACD line < signal line → блокировать лонг. calcMACD уже есть в core.js:278. ~5 строк в filter_registry.js.

---

**30. Efficiency Ratio (ER) фильтр**
*[arXiv: Assessing the Impact of Technical Indicators — adaptive indicators section]*

Perry Kaufman's Efficiency Ratio: `ER = |price_change_N_bars| / sum(|bar_changes|)`. ER близкий к 1 = сильный тренд, ER близкий к 0 = хаос/боковик. Фильтр: входить только при ER > 0.3 (движение эффективно). Адаптируется к рынку лучше ADX потому что учитывает "прямолинейность" движения. ~10 строк.

### 🛡️ ГРУППА 5 — Stop-Loss и Position Sizing (из поиска Kelly/Drawdown)

---

**31. Оптимальный Stop-Loss через Bayesian Analysis**
*[arXiv: Determining Optimal Stop-Loss Thresholds via Bayesian Analysis of Maximum Drawdown — arXiv:1609.00869]*

Статья предлагает выбирать SL не произвольно, а через байесовскую оценку: при каком уровне SL математическое ожидание Maximum Drawdown минимально? Формула зависит от исторических параметров стратегии. Для USE Optimizer: добавить режим "auto SL" который рассчитывает оптимальный SL множитель из бэктест результатов. Не оптимизация по сетке, а аналитическое решение.

---

**32. Kelly Fraction с ATR-коррекцией**
*[arXiv: Type of the Paper (Kelly Criterion) — arXiv:1710.00431]*

Классический Kelly: `f = (p×b − q) / b` где p = win rate, b = avg win / avg loss. ATR-коррекция: уменьшать Kelly при высоком ATR (высокая волатильность = рискуй меньше). `f_atr = f × (atr_avg / atr_current)`. Уже есть `tradePnl[]` для расчёта p, b. Добавить как рекомендацию в detail-модале: "рекомендуемый размер позиции по Kelly = X%".

---

**33. Fractional Kelly (половина Kelly)**
*[arXiv: Risk averse fractional trading using the current drawdown — arXiv:1612.02985]*

Статья математически доказывает: `f/2` (половина Kelly) оптимально для risk-averse трейдера с дравдаун-ограничением. Полный Kelly максимизирует рост но создаёт огромные просадки. Половина Kelly = ~75% роста полного Kelly + просадки в 4 раза меньше. Добавить в detail-модале: "Full Kelly: X%, Half Kelly: Y%, Quarter Kelly: Z%".

---

**34. Trailing Stop по текущей просадке**
*[arXiv: Risk averse fractional trading using the current drawdown]*

Адаптивный trailing stop: чем больше текущая просадка от пика equity → тем ближе стоп. Если просадка 0% → стандартный стоп. Если просадка 5% → стоп в 1.5× ближе. Если просадка 10% → уменьшить позицию вдвое. Это "equity trailing stop" — защищает капитал на уровне серии сделок, не отдельной сделки. В core.js: передавать текущий pnl/dd в backtest для адаптивного стопа.

---

**35. VIX-ранговое позиционирование**
*[arXiv: Sizing the Risk: Kelly, VIX, and Hybrid Approaches — arXiv:2508.16598]*

При высоком VIX (страх) — уменьшать размер позиции. VIX-rank = процентиль текущего VIX за последние 252 дня. `size = 1 − VIX_rank`. Прямо из статьи: гибридный подход Kelly×VIX outperforms каждый по отдельности. Применительно к USE Optimizer: добавить поле "VIX data" (отдельный CSV) и фильтр по VIX-rank.

---

**36. Максимальный допустимый размер позиции из просадки**
*[arXiv: Drawdown: From Practice to Theory and Back Again — arXiv:1404.7493]*

Статья даёт теорему: при заданной вероятности разорения p и исторической волатильности, максимальный безопасный размер позиции = f_max. Если трейдер задаёт "максимально допустимый риск 5% от депо" — автоматически вычислить SL множитель. Это обратная задача к Kelly: не "сколько зарабатываем" а "сколько можем потерять".

---

**37. Compound Annual Growth Rate (CAGR) расчёт**
*[arXiv: Performance metrics table — arXiv:2405.05449]*

Простая но отсутствующая метрика: `CAGR = (1 + total_return) ^ (1/years) − 1`. Нужно знать число лет в данных = `N_bars / bars_per_year`. Bars_per_year: пользователь указывает таймфрейм (D/H4/H1/etc) и система вычисляет. Добавить выпадающий список таймфрейма в UI. Тогда PnL% можно аннуализировать для сравнения между разными ТФ.

### 🔮 ГРУППА 6 — Режимы рынка (из поиска по режимам/HMM)

---

**38. Hidden Markov Model (HMM) детектор режимов**
*[arXiv: A Hybrid Learning Approach to Detecting Regime Switches in Financial — q-fin]*

HMM с 2-3 скрытыми состояниями (бычий/медвежий/боковой) обучается на исторических доходностях и волатильности. Даёт вероятность текущего режима. Применение в USE Optimizer: добавить HMM как дополнительный фильтр — блокировать лонги при P(медвежий режим) > 0.7. Нужна реализация Baum-Welch (~100 строк JS) или упрощённый порог по returns/vol ratio.

---

**39. Трёхсостояниевый Markov filter (Bull/Normal/Crisis)**
*[arXiv: When the Rules Change: Adaptive Signal Extraction — Three-State Markov Switching]*

Статья использует 3 режима: Bull (растущий), Normal, Crisis (кризис с jumps). Каждый режим имеет свои параметры доходности и волатильности. Практическое приложение: детектировать "Crisis" режим и выключать стратегию полностью (stay flat). Упрощённая реализация без HMM: если rolling_vol > 3× исторической median_vol → Crisis → не торгуем.

---

**40. Realized Covariance для детекции режимов**
*[arXiv: Market Regime Detection via Realized Covariances — arXiv:2104.03667]*

Статья использует realised covariance (RCov) между активами для детекции смены режимов. Для ONE актива: realised variance = sum(bar_returns²) за окно. Высокий RVar = нестабильный режим. Добавить как фильтр: `useRVar` = блокировать сигналы когда realized variance выше N-ого персентиля. Работает без данных объёма, только OHLC.

---

**41. Adaptive Parameter Switching по режиму**
*[arXiv: When the Rules Change — adaptive signal extraction]*

Идея: хранить ДВА набора параметров — "тренд-параметры" и "боковик-параметры". В зависимости от детектированного режима (ADX > 25 = тренд) использовать соответствующие параметры. В оптимизаторе: оптимизировать оба набора одновременно и выбирать по ADX. Это фундаментально новый режим работы стратегии.

---

**42. Nonlinear Kalman Filter для адаптивных параметров**
*[arXiv: Nonlinear Filters for Hidden Markov Models — q-fin]*

Kalman Filter обновляет параметры (например период MA) в реальном времени по мере поступления новых данных: параметры "дрейфуют" вместе с рынком. Адаптивная MA: период сжимается при высокой волатильности и расширяется при низкой. ~30 строк JS. Тестировать: даёт ли адаптивная MA лучший IS/OOS чем фиксированный период?

## 50 задач для USE Optimizer

Разбиты по категориям: метрики, оптимизация, валидация, сигналы, UI, данные, инфраструктура, агент.

### 📊 НОВЫЕ МЕТРИКИ (продолжение серии UPI/Sortino/K-Ratio)

---

**1. Profit Factor**
Самая известная метрика трейдинга: сумма всех выигрышных сделок делённая на сумму всех проигрышных. PF > 1.5 = хорошо, > 2 = отлично. Уже есть `tradePnl[]` в core.js — буквально 5 строк. Большинство трейдеров смотрят именно на неё, и её отсутствие в таблице — заметный пробел.

---

**2. Recovery Factor**
Сколько раз стратегия "отбила" свой максимальный убыток: `pnl / maxDD`. Если заработали 50% и максимальная просадка была 10% — Recovery Factor = 5. Отличается от Calmar тем, что Calmar = годовой PnL / DD, а Recovery Factor = полный PnL / DD. Проще для понимания. ~3 строки в `opt.js`.

---

**3. Expectancy (математическое ожидание на сделку)**
`avgWin × winRate − avgLoss × (1 − winRate)`. Показывает сколько % в среднем зарабатывает каждая сделка с учётом и побед и поражений. Стратегия с WR 40% может быть прибыльной если avgWin большой. Очень полезна как фильтр: expectancy < 0 = убыточная математически. ~5 строк.

---

**4. Max Consecutive Losses**
Максимальная серия убыточных сделок подряд. Психологически важнейшая метрика — большинство трейдеров бросают стратегию именно из-за длинной серии поражений, а не из-за общей просадки. Нужен `tradePnl[]`, простой цикл. Добавить как колонку и фильтр.

---

**5. Average Bars in Trade**
Среднее количество баров, которое стратегия держит позицию. Нужно добавить счётчик в `backtest()` при закрытии сделки. Полезно: стратегия с коротким удержанием и высоким числом сделок имеет другой профиль риска чем долгосрочная. Помогает понять временной горизонт стратегии.

---

**6. Time in Market %**
Какой процент времени есть открытая позиция. `(суммарные бары в сделках / общее число баров) × 100`. Стратегия с 5% времени в рынке при тех же метриках лучше стратегии с 80% — меньше рисков. Добавить счётчик баров в `backtest()`. ~5 строк.

---

**7. Sharpe Ratio (из equity curve)**
Классика финансов: `среднее изменение equity / стандартное отклонение`. Уже есть все данные в `eq[]`. Отличается от Sortino тем, что штрафует и за позитивную волатильность — некоторые считают это недостатком, другие — преимуществом. Нужен для полноты набора метрик.

---

**8. Calmar Ratio (годовой)**
Текущий `pdd` это `pnl/dd`, но классический Calmar = **годовой** PnL / maxDD. Если добавить расчёт числа лет в данных — можно получить настоящий Calmar, сопоставимый с внешними системами. Нужно знать таймфрейм или количество баров в году.

---

**9. Tail Ratio**
Правый хвост прибыли к левому хвосту убытков: `95-й персентиль доходности / |5-й персентиль|`. Показывает асимметрию: хорошая стратегия должна иметь большие выигрыши и маленькие проигрыши. Уже есть `tradePnl[]`, нужна сортировка + персентили. ~10 строк.

---

**10. Gain to Pain Ratio (GPR)**
`суммарный PnL / сумма абсолютных значений всех отрицательных периодов equity`. Придуман Stevem Cohen. Похож на Omega (задача 179 в opt.js уже есть), но считается по-другому — через equity bars, а не increments. Сравнение двух подходов само по себе интересно.

---

### 🔍 ВАЛИДАЦИЯ И ROBUSTNESS

---

**11. Sensitivity Heatmap (карта чувствительности)**
Берём лучший результат, фиксируем все параметры, меняем два любых параметра по сетке ±20% и рисуем 2D карту цветом (зелёный = хороший PnL, красный = плохой). Визуально показывает "остров стабильности" — хорошая стратегия должна быть прибыльной в широком диапазоне вокруг оптимума, а не только в одной точке.

---

**12. Anchored Walk-Forward**
Сейчас IS/OOS делается один раз (70%/30%). Anchored WF: IS всегда начинается с начала данных и постепенно расширяется (60%, 65%, 70%, 75%), OOS = следующий кусок. Результат — несколько OOS периодов, можно посмотреть деградацию по времени. Лучше выявляет стратегии с "распадом альфы".

---

**13. Rolling Walk-Forward**
В отличие от anchored — окно IS фиксированного размера скользит вперёд. Например IS = 500 баров, OOS = 100 баров, шаг = 50 баров. Получаем серию OOS-результатов и смотрим насколько они стабильны. Стандартный подход в алго-трейдинге для отбора стратегий.

---

**14. Parameter Stability Score**
Запустить оптимизацию на N случайных "шумовых" вариантах параметров (±5-10% от оптимума). Считать % вариантов с положительным PnL. Если 80%+ — стратегия стабильна. Разновидность robustness test, но быстрее и нагляднее чем текущий Mass Robust.

---

**15. Bootstrapped Equity Curve**
Берём `tradePnl[]`, случайно перемешиваем 1000 раз (уже есть MC Permutation), но вместо p-value строим **диапазон** equity curves: медиана, 5-й и 95-й персентили. Рисуем на графике серым коридором. Пользователь видит разброс возможных траекторий при той же стратегии. Визуально убедительно.

---

**16. Regime-Based Validation**
Разбить DATA на периоды по характеру рынка: трендовый (ADX > 25), боковой (ADX < 20), высокая волатильность (ATR > среднего), низкая волатильность. Запустить бэктест на каждом режиме отдельно. Показать в detail-модале: "В тренде: +15%, В боковике: -3%". Уже есть все нужные индикаторы.

---

**17. IS vs OOS Degradation Score**
Формализовать "насколько OOS хуже IS": `(IS_pnl/bar − OOS_pnl/bar) / IS_pnl/bar`. Если OOS = 70% от IS — хороший результат. Если OOS = 20% — overfitting. Добавить как отдельную метрику в таблицу и фильтр. Данные уже есть в `cfg._oos.forward`.

---

**18. Overfitting Index (OFI)**
Количество тестируемых параметров делённое на количество сделок: `n_params / n_trades`. Bailey et al. показали, что OFI > 0.3 означает высокую вероятность overfitting. Простое предупреждение пользователю: "Внимание: слишком много параметров на количество сделок".

---

**19. Deflated Sharpe Ratio (DSR)**
Корректировка Sharpe с учётом того, что мы перебирали много вариантов. Чем больше комбинаций перебрано — тем выше надо поднять планку для Sharpe. Bailey & López de Prado 2014. Показывает истинную значимость результата с учётом multiple testing bias.

---

**20. Asymmetry Score (Long vs Short качество)**
Уже есть `wrL`, `wrS`, `dwrLS`. Задача — расширить: посчитать отдельные метрики (PnL, DD, Sharpe) для лонгов и шортов и показать в detail-модале. Помогает понять: стратегия работает только в одну сторону или в обе? Данные уже частично есть (`nL`, `pL`, `nS`, `pS`).

---

### ⚡ ОПТИМИЗАЦИЯ И ПОИСК

---

**21. Simulated Annealing**
Альтернатива TPE и HC. Алгоритм "охлаждения": сначала принимает случайные ухудшения (высокая "температура"), постепенно становится строже. Хорошо избегает локальных максимумов. Классика для задач оптимизации параметров. Реализуется как новый режим рядом с TPE/MC в `runOpt()`.

---

**22. Multi-Objective Optimization (Pareto Front)**
Сейчас оптимизируем по одной метрике (PnL или GT-Score). Идея: оптимизировать по двум метрикам одновременно (например PnL и Robustness) и находить **фронт Парето** — набор решений, где улучшение одной метрики ухудшает другую. Пользователь видит trade-off и выбирает точку сам.

---

**23. Parameter Importance (Feature Importance)**
После exhaustive-оптимизации посчитать корреляцию каждого параметра с итоговым PnL. Нарисовать bar chart: "SL множитель влияет на 40% результата, RSI порог — на 5%". Пользователь понимает что оптимизировать тщательно, а что можно не трогать. Метод: ANOVA или просто ранговая корреляция.

---

**24. Adaptive Grid Refinement**
Первый проход — грубая сетка (шаг 0.5). Находим топ-20% результатов, делаем второй проход вокруг них с мелким шагом (0.1). Третий — ещё мельче. Экономит время: вместо сетки 10×10×10=1000 точек делаем 5×5×5 + 3×3×3 + ... = значительно меньше при той же точности.

---

**25. Ensemble Score**
Взять топ-5 параметрических наборов и посчитать их "среднюю" equity curve. Если ансамбль показывает лучший Sharpe, чем любой отдельный — это признак надёжности. Идея из ML: ансамблевые методы устойчивее отдельных моделей. Реализация: просто усреднить equity curves.

---

**26. Bayesian Optimization (BO)**
TPE уже является разновидностью BO, но классический BO с гауссовским процессом более точен в маленьких размерностях. Идея: заменить модель TPE на GP для параметрических пространств с ≤ 5 параметрами и переключаться автоматически. GP лучше моделирует неопределённость и точнее направляет поиск.

---

**27. CMA-ES (Evolution Strategy)**
Covariance Matrix Adaptation — один из лучших алгоритмов для непрерывной оптимизации. Учит форму "яйцевидной" области хороших параметров и концентрирует поиск там. Особенно хорош когда параметры коррелированы (например SL и TP обычно связаны). Готовые реализации компактны (~100 строк).

---

**28. Stochastic Grid с перестановкой**
Вместо перебора всех комбинаций (exhaustive) — случайно перемешать порядок итерации по сетке. Дать пользователю смотреть результаты "в прямом эфире" в случайном порядке. При прерывании оптимизации нет систематического смещения (сейчас пользователь видит только первые значения диапазона).

---

### 🎯 НОВЫЕ СИГНАЛЫ ВХОДА И ФИЛЬТРЫ

---

**29. Bollinger Band Squeeze Entry**
Вход после периода сжатия волатильности: BB bandwidth (расстояние между верхней и нижней полосой) падает ниже порога, потом цена выходит за полосу. Классика для торговли пробоев после консолидации. Новый entry в `entry_registry.js`, BB легко вычисляется через SMA + стандартное отклонение.

---

**30. MACD Crossover Entry**
MACD line пересекает signal line снизу вверх (лонг) или сверху вниз (шорт). Один из самых популярных сигналов в мире. calcMACD уже есть в `core.js:278`. Нужно только добавить entry в реестр, который проверяет пересечение на предыдущем баре.

---

**31. Stochastic Crossover Entry**
%K пересекает %D в зоне перепроданности (< 20) или перекупленности (> 80). calcStochastic уже есть в `core.js:315`. Хорошо сочетается с существующим RSI-фильтром и дополняет его — другие математические свойства.

---

**32. Inside Bar (IB) Entry**
Свеча, которая полностью помещается внутрь предыдущей (high < prev.high, low > prev.low). Вход при пробое максимума или минимума IB. Паттерн сжатия — похож на BB Squeeze но на свечном уровне. Простой detectL/detectS, ~10 строк.

---

**33. Morning/Evening Star (3-свечной паттерн)**
Классический японский паттерн разворота: большая медвежья свеча → маленькая (дожи/звезда) → большая бычья. Реализация: проверить 3 свечи назад. Сложнее Engulfing но мощнее на дневных данных.

---

**34. Volatility Breakout (Donchian Channel)**
Вход при пробое N-дневного максимума (лонг) или минимума (шорт). Основа систем тренд-следования типа "Черепах" Ричарда Денниса. Прост в реализации: `max(high[1..N])`, ~5 строк. Интересно проверить в сочетании с существующими фильтрами ADX и Structure.

---

**35. Rate of Change (RoC) Filter**
Фильтр: входить только если скорость изменения цены за N баров выше порога. `(close - close[N]) / close[N] × 100`. Отсекает "вялые" движения, оставляет только сильные импульсы. Простой blocksL/blocksS, ~5 строк. Дополняет ATR Expanding (задача уже есть как `atrexp`).

---

**36. CCI (Commodity Channel Index) Filter**
`(price - SMA) / (0.015 × meanDeviation)`. CCI > 100 = перекупленность, < -100 = перепроданность. Использовать как фильтр: не входить в лонг при CCI > 200 (уже перегрето). Интересная альтернатива RSI с другим математическим поведением.

---

**37. Volume Profile / VWAP фильтр**
VWAP (Volume Weighted Average Price): средняя цена взвешенная по объёму. Многие институциональные трейдеры используют его как ключевой уровень. Входить только когда цена выше/ниже VWAP. Требует данных объёма (`HAS_VOLUME`). Активируется только если объём доступен.

---

**38. Supertrend Entry**
Вход при смене направления Supertrend: с медвежьего на бычий (лонг) или наоборот (шорт). calcSupertrend уже есть в `core.js:289`. Нужно только entry в реестр, который проверяет `dir[i-1] !== dir[i-2]`. Одна из лучших трендовых систем.

### 11. Risk of Ruin

**Что нашлось (backtestbase.com, quantifiedstrategies, myfxbook, tradingindepth):**

**Формула (backtestbase):**
```
RoR = ((1 - Edge) / (1 + Edge))^(Capital/RiskPerTrade)
где Edge = WR - (1-WR)/RR
```
Или через MC simulation (tradingindepth делает Monte Carlo + фиксированный порог).

**Ключевые пороги:**
- Professional traders: RoR < 5%
- Institutional funds: RoR < 1%
- "Loss level" обычно 25-50% drawdown (не full ruin)

**У нас всё для этого есть:** WR, avgWin/avgLoss — 3 строки JS. Показывать рядом с Kelly.

**Сложность:** ★ | **Ценность:** ★★★★★

### 13. Streak Analysis (серии побед/поражений)

**Что нашлось (TradesViz, alphaexcapital, TORtrading):**

**TradesViz (конкурент) — формулы:**

MaxConsecutiveWins = длина longest win streak
MaxConsecutiveLosses = длина longest loss streak
AvgConsecutiveWins / AvgConsecutiveLosses
ExpectedMaxLossStreak = log(N) / log(1/WR)  // математическое ожидание

**Ключевые инсайты:**

- Психологическое значение: нужно знать worst-case streak заранее чтобы не бросить стратегию

**Сложность:** ★ | **Ценность:** ★★★★

### 14. Alpha / Beta / Information Ratio

**Что нашлось (Investopedia, FasterCapital, AnalystPrep):**

**Формулы:**
```
α (Jensen's Alpha) = R_strategy - [Rf + β(R_market - Rf)]
β = Cov(R_strat, R_market) / Var(R_market)
Information Ratio = (R_strat - R_benchmark) / TrackingError
```
Где TrackingError = std(excess daily returns).

**Проблема для нашего проекта:**
Нам нужны данные о benchmark (цена актива как buy&hold). У нас есть equity curve стратегии, НО нет buy&hold equity curve — её нужно считать отдельно при бэктесте.

**AnalystPrep** описывает целый блок метрик: Sharpe, Treynor, Jensen's Alpha, Information Ratio, Modigliani-squared (M²).

**Сложность:** ★★★ (нужен benchmark в движке) | **Ценность:** ★★★★

### 16. CAGR (Compound Annual Growth Rate)

**Что нашлось (Investopedia, quantifiedstrategies, Ryan O'Connell CFA):**

**Формула:**
```
CAGR = (FinalEquity/InitialEquity)^(1/Years) - 1
```

**Ryan O'Connell (2026):**
> "CAGR, geometric mean, arithmetic mean — разные вещи. CAGR = geometric mean = правильный способ считать annualized return для сложного процента"

**Почему важно:**
- 100% return за 2 года ≠ 50% годовых (arithmetic) = 41.4% CAGR (geometric)
- CAGR/MaxDD — популярный composite ratio (MAR ratio)
- У нас есть equity curve и timestamps → CAGR считается в одну строку

**Сложность:** ★ | **Ценность:** ★★★★★ — фундаментальная метрика, которой у нас нет!

### 17. VaR / CVaR (Value at Risk / Conditional VaR)

**Что нашлось (Investopedia, QuantInsti, GARP whitepaper):**

**VaR (95%)** = потеря, которую стратегия не превысит в 95% случаев
```
VaR_95 = quantile(daily_returns, 0.05) * capital
```

**CVaR / Expected Shortfall** = среднее потерь в худшем 5% (хвост):
```
CVaR_95 = mean(returns[returns < VaR_threshold])
```

**QuantInsti:**
> "VaR говорит 'не хуже X в 95% случаев', но молчит о том что будет в оставшихся 5%. CVaR отвечает на этот вопрос."

**Нужны данные:** per-bar equity returns (уже есть).

**Сложность:** ★★ | **Ценность:** ★★★★ — институциональный стандарт

---

### 18. Returns Distribution (Гистограмма + Skewness/Kurtosis)

**Что нашлось (GitHub fderyckel, DataCamp, BMC Medical):**

**fderyckel (GitHub Pages):**
> "Финансовые returns НЕ нормально распределены — это fat tails и отрицательный skew. Jarque-Bera test проверяет normality."

**DataCamp:**
```
Skewness = E[(X-μ)³] / σ³       // <0 = больше убытков, >0 = больше прибылей
Kurtosis = E[(X-μ)⁴] / σ⁴ - 3  // >0 = fat tails (экстремальные выбросы)
Jarque-Bera = n/6 * (S² + K²/4)  // H0: нормальное распределение
```

**Применение:**
- Гистограмма per-trade PnL (уже делаем частично)
- Добавить Skewness и Kurtosis как числа
- Jarque-Bera p-value → "распределение нормальное / имеет fat tails"

**Сложность:** ★★ | **Ценность:** ★★★★ — объясняет почему Sharpe может лгать

---

### 19. ATR-Based Position Sizing

**Что нашлось (quantstrategy.io, alphaexcapital, volatilitybox, LuxAlgo):**

**Формула (volatilitybox, 2026):**
```
PositionSize = (Capital × RiskPct) / (ATR × StopMultiplier)
// Например: 100k × 1% / (ATR14 × 2) = 500 / ATR
```

**Идея Marcos Lopez de Prado (setup4alpha):**
> "Volatility-adjusted sizing вместо fixed lots — не перегружаться в волатильные периоды, не недогружаться в тихие"

**Применение к нашему проекту:**
В UI показывать: "при ATR=X и риске 1% на сделку — рекомендуемый размер позиции = Y контрактов/лотов". Требует знать ATR актива и размер капитала — можно как input.

**Сложность:** ★★ | **Ценность:** ★★★★ — практически очень ценно

## 10 новых исследовательских задач

### 1. **Omega Ratio** ⭐ Метрика (лёгкая)
`Ω = Σmax(Δeq,0) / Σmax(-Δeq,0)` — профит-фактор на уровне баров.
Уже упомянут в отчёте 2026-03-03 как «наиболее полная метрика — учитывает всё распределение». Только `equity[]`, ~10 строк.
Отличие: в отличие от UPI/Sortino не нормализован на PnL — это «сколько зарабатывает на 1 рубль потерь».

---

### 2. **MC Permutation Test** ⭐⭐ Валидация (средняя)
Перемешать порядок сделок 1000x → считать долю случайных equity ≥ actual → p-value.
Зависит от задачи SQN (per-trade array в core.js). Высокая научная ценность: отличает «скилл» от «удачи последовательности». Единственный способ получить статистически корректный p-value для стратегии.

---

### 3. **Parameter Landscape Heatmap** ⭐⭐⭐ UI/UX (большая, очень ценная)
После оптимизации — 2D тепловая карта двух ключевых параметров. Цвет = PnL/GT-Score.
Позволяет **визуально** видеть overfitting: острые пики → overfitted, плавные холмы → устойчиво.
Реализация: после runOpt запустить exhaustive scan по 2 выбранным param, рендерить `<canvas>`.

---

### 4. **Multi-objective Pareto Front** ⭐⭐⭐ Алгоритм (крупная)
Оптимизировать одновременно по 2 целям (например, PnL и robScore). Отображать фронт Парето — несравнимые решения.
Сейчас: один скор → одно «лучшее». С Парето: пользователь видит трейдоф и выбирает точку на фронте сам.

---

### 5. **Rolling Walk-Forward (WFO)** ⭐⭐ Валидация (средняя)
Вместо фиксированного 70/30 — скользящее окно (например, 5 окон по 20%).
Метрика устойчивости = % окон с положительным OOS. Показывает «умирает ли стратегия» со временем.
Дополняет CPCV — проще интерпретировать трейдерам.

---

### 6. **MAE / MFE Analysis** ⭐⭐ Per-trade (средняя)
Maximum Adverse Excursion / Maximum Favorable Excursion — требует данных по каждой сделке.
MAE показывает насколько сделка уходит «против» перед закрытием → оптимальное расположение стоп-лосса.
MFE → оптимальный тейк-профит. Scatter plot MAE vs результат сделки = классический инструмент трейдера.

---

### 7. **CMA-ES Optimizer** ⭐⭐ Алгоритм (большая)
Covariance Matrix Adaptation Evolution Strategy — элитный эволюционный оптимизатор для непрерывных пространств.
Лучше TPE для высокоразмерных задач (>5 параметров). Готовые JS-реализации минимальны (~200 строк).
Особенность: адаптирует форму мутаций под ландшафт → быстрее находит оптимум.

---

### 8. **Kelly / Half-Kelly Position Sizing Simulator** ⭐ Симуляция (лёгкая)
`f* = (WR * avgWin - (1-WR) * avgLoss) / avgWin` — оптимальная доля капитала под риск.
Показать в detail-модале: Kelly %, Half-Kelly %, симулированный рост $1000 с этим sizing.
Трейдеры применяют Kelly ежедневно — наглядный практический вывод из метрик.

---

### 9. **Strategy Correlation Matrix** ⭐⭐ UI (средняя)
При наличии нескольких результатов — матрица корреляции equity curves (Pearson/Spearman).
Низкая корреляция = хороший портфель стратегий. Цветная heatmap в отдельной вкладке.
Позволяет пользователю строить некоррелированные портфели прямо в браузере.

---

### 10. **Smart Cluster Detection в результатах** ⭐⭐ Анализ (средняя)
k-means или DBSCAN по пространству параметров → группировать результаты в кластеры.
Цель: найти «острова» параметров с похожей производительностью → самый большой остров = наиболее устойчивый регион.
Вместо выбора одной лучшей точки — выбрать центр наиболее плотного/большого кластера.
