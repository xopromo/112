# Архитектурные решения USE Optimizer

## ✅ Решение: JS-only бэктест (без серверных вычислений)

**Статус**: Принято и реализовано
**Причина**: offline-first, максимальная скорость для веб-приложения
**Файлы**: `core.js:378` (backtest), `opt.js` (оптимизация)

**Отклонено**:
- WebWorkers — слишком сложная синхронизация
- WASM (Rust) — в очереди, но не приоритет сейчас
- Node.js backend — усложняет деплой на GitHub Pages

---

## ✅ Решение: Pine Script v6 экспорт

**Статус**: Активно используется
**Файл**: `pine_export.js:19` (generatePineScript)
**Причина**: v5 deprecated, v6 — актуальный стандарт TradingView

**Реализация особенностей**:
- `_addActivePinev6()` — toggle-группы с `active=` переменной (строка 1376)
- 14 групп параметров с зависимостями
- Автоисправление ошибок (`fixPineScript`, строка 1260)

**Отклонено**:
- Pine v5 — старая версия
- Экспорт на другие платформы — только TV на данный момент

---

## ✅ Решение: Двухуровневая валидация (IS/OOS split)

**Статус**: Реализовано, но требует улучшений
**Файл**: `opt.js:941` (_attachOOS)
**Цель**: Борьба с переоптимизацией

**Как работает**:
- IS (In-Sample) = первые 70% данных
- OOS (Out-of-Sample) = остаток данных
- Метрика Retention = OOS_PnL / IS_PnL (должна быть близка к 1)
- TV-сравнение показывает реальность vs JS расчёты

**Известная проблема**:
- HC (Hill Climbing) результаты НЕ имеют IS/OOS split
- Требуется реализовать `_hcBuildOOS()` для соседей

**Отклонено**:
- Simple 50/50 split — слишком мало данных в OOS
- Walk-forward (5+ блоков) — слишком медленно для UI

---

## ✅ Решение: Исследовательский агент (3-уровневый pipeline)

**Статус**: Работает по крону каждый час
**Файлы**: `agent/research_pipeline.sh` (главный), `agent/groq_helper.sh`

**3 уровня**:
1. **Groq (llama-3.3-70b)** — бесплатно, быстро, 6 URL источников
2. **Haiku** (~$0.01) — фильтрует к 1-2 гипотезам
3. **Sonnet** (~$0.10, если есть план) — реализует код

**Экономия**: 85% токенов vs прямого вызова Sonnet

**Отклонено**:
- Одноуровневый агент (Sonnet) — дорого
- Ручное управление — ненадежно

---

## ✅ Решение: Метрики качества вместо просто PnL

**Статус**: Частично реализовано
**Реализованные**:
- Sig% — z-тест WR > 50% (opt.js:43)
- GT-Score — anti-overfitting метрика (opt.js:58)
- UPI — Ulcer Performance Index (opt.js с eq[])
- K-Ratio — линейность роста (вычисляется но не экспортируется)

**В очереди** (research_reports/2026030320.md):
- Sortino Ratio — only downside volatility
- SQN (System Quality Number) — требует per-trade array
- Omega Ratio, Pain Index

**Отклонено**:
- Sharpe (требует benchmark) — не подходит для стратегий
- CAGR только — упускаетDD важность

## ⚠️ Решение: Одна табличка результатов (не по рынкам/периодам)

**Статус**: Текущая реализация
**Причина**: Упрощение UI, быстрый прототип

**Отклонено**:
- Multi-market сравнение — требует структурировать DATA по инструментам
- Сегментация по периодам — усложняет бэктест

**Планируется**:
- Когда появятся стандартные наборы данных, добавить фильтрацию

## 🔴 Известный баг: TV vs JS расхождения

**Статус**: Активная диагностика
**Причины**:
- Неправильный расчёт SL/TP при combo режиме
- Прогрев MA обработан по-разному
- Цена входа может вычисляться по close[bar] vs open[bar+1]

**Как диагностировать**:
1. Загрузить TV CSV из индикатора USE_EXP
2. Сравнить equity% построчно
3. Найти первое расхождение > 0.5%
4. Проверить конкретный бар в логике SL/TP

## 📋 Правило: Полнота _cfg объектов

**Статус**: Критично соблюдать
**Файл**: CLAUDE.md "Правило: полнота _cfg объектов"

Каждый фильтр/параметр в inline `btCfg` ОБЯЗАН присутствовать в 3 местах:
- `_cfg` (MC, opt.js ~1909)
- `_cfg_tpe` (TPE, opt.js ~2265)
- `_cfg_ex` (Exhaustive, opt.js ~2847)

Если поле отсутствует → OOS прогон использует другие параметры → неправильные TV-метрики.

## 📋 Правило: Прогрев индикаторов (warmup)

**Статус**: Критично соблюдать при новых фильтрах
**Файл**: filter_registry.js

**Проблема**: JS инициализирует Float64Array нулями → первые period-1 баров = 0
**Pine**: возвращает na → сравнение с na = false → сигнал заблокирован

**Решение**: Если `indicatorArr[i-1] <= 0` → **блокировать сигнал**

**Исправлено в**: фильтры `ma`, `confirm`, `strend` (commit 991ebaf)
