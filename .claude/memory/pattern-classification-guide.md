# Pattern Classification Guide: CLASS A vs CLASS B

## Проблема: Amnesia-Driven Development

```
Мы исправляем Float32Array в 22 местах
    ↓
Проблема исчезает (хорошо!)
    ↓
Но она больше никогда не повторяется (потому что мы её исправили)
    ↓
Новый разработчик делает похожую ошибку в НОВОМ месте
    ↓
"Мне кажется я это видел..." ← никто не помнит где
    ↓
Лазаем по коду, ищем баги... СНОВА
```

**Решение:** Не ждать повторения - определить что это ПАТТЕРН СРАЗУ и обобщить!

---

## Decision Tree: CLASS A vs CLASS B

```
НАЙДЕН БАГ
    │
    ├─ Есть ли ПОХОЖИЕ места в коде? (grep поиск)
    │  │
    │  ├─ ДА (≥2 места)
    │  │  └─ → CLASS A: СИСТЕМНЫЙ ПАТТЕРН
    │  │
    │  └─ НЕТ (только 1 место)
    │     │
    │     └─ Может ли эта ошибка произойти в АРХИТЕКТУРНО похожих местах?
    │        │
    │        ├─ ДА (теоретически возможна в других файлах/функциях)
    │        │  └─ → CLASS A: Потенциальный паттерн (обобщить!)
    │        │
    │        └─ НЕТ (уникально для этого контекста)
    │           └─ → CLASS B: Случайная ошибка (просто исправить)
```

---

## Критерии CLASS A: СИСТЕМНЫЕ ПАТТЕРНЫ

**ОБЯЗАТЕЛЬНО обобщить в универсальное правило!**

### Критерий 1: Architectural Consistency (следует архитектуре)
Ошибка возникает потому что архитектура требует этого, а не потому что невнимательность.

```javascript
// CLASS A: Архитектурная ошибка
// Везде где Array передаётся → может быть то же нарушение
obj.field = source.field;  // Reference instead of copy

// CLASS B: Невнимательность
// Опечатка в одном месте, не архитектурная
const nLenght = arr.length;  // typo, не в архитектуре
```

**Вопрос:** "Это следует паттерну архитектуры или это невнимательность?"

### Критерий 2: Repeatable Across Contexts (повторима в других контекстах)

```javascript
// CLASS A: Может повториться
// Новый файл ui_future.js может сделать то же
rNew.eqCalcMAArr = _btCfg.eqCalcMAArr;  // Reference!

// CLASS B: Уникально
// Эта переменная только в этом месте
let result = calculate(input);  // не будет в других местах
```

**Вопрос:** "Может ли эта ошибка произойти в архитектурно похожих местах?"

### Критерий 3: Systematic Solution (требует универсального решения)

```javascript
// CLASS A: Нужно одно правило для всех
// Copy-on-Storage Pattern покроет ВСЕ Array/Object в любом файле
if (cfg.eqCalcMAArr) btCfg.eqCalcMAArr = Array.from(cfg.eqCalcMAArr);

// CLASS B: Уникальный фикс
// Опечатку в одном месте нельзя "обобщить"
typo = "nLenght" → "length"  // просто исправить
```

**Вопрос:** "Одно правило покроет все случаи или нужны уникальные фиксы?"

## Как определить автоматически (в голове):

**Задай себе три вопроса:**

### Вопрос 1: "Это в одном месте или я вижу похожее ещё?"
```
grep -r "то же нарушение" opt.js ui_*.js

Результат:
- ≥2 места → CLASS A
- 1 место  → Вопрос 2
```

### Вопрос 2: "А может, это скрывается в архитектуре и повторится потом?"
```
Примеры CLASS A в одном месте:
- "Забыл Array.from() при сохранении Array"
  → Это паттерн! Может повториться в новом коде
  
- "Неправильное имя переменной (typo)"
  → Это невнимательность, не архитектура
```

**Вопрос себе:** "Если я напишу новый файл ui_future.js, повторится ли эта ошибка?"
- Если ДА → CLASS A (даже если сейчас 1 место)
- Если НЕТ → CLASS B

### Вопрос 3: "Одно правило покроет ВСЕ или нужны разные решения?"
```
CLASS A:
  Copy-on-Storage Pattern → Array.from() везде
  Rule: "При сохранении Array → копировать"
  Одно правило, все места
  
CLASS B:
  "Переименовать переменную"
  Нет универсального правила, уникальный фикс
```
